How I used AI in this project

This page explains how AI tools supported my work, what they helped me produce, what I changed manually, what limitations I noticed, and how I critically evaluated reliability, risks, and ethics while developing the project.

Overview

Why this reflection matters

AI tools were helpful during this project because they supported both the technical and explanatory sides of development. They helped me generate draft code, improve layout ideas, add comments, expand explanations, and fix some implementation issues more quickly than I could have done alone as a beginner. However, using AI did not remove the need for my own understanding. In several cases, the generated output still needed testing, editing, and correction before it became suitable for the assignment.

The most important lesson I learned is that AI is useful as a support tool, not as a full replacement for careful thinking. Some responses were helpful immediately, while others were incomplete, too generic, or introduced mistakes when copied into the project. Because of this, I had to actively evaluate the output instead of assuming it was always correct.

In this project, AI was most helpful when I already knew the goal I wanted to achieve. For example, it was useful when I asked for a styled tutorial page, a dynamically loaded quiz, better comments, or more detailed explanations. It was less reliable when the task required exact alignment with the assignment brief or when copy-paste errors caused JavaScript to break. This meant I had to review both the code and the content carefully after every major change.

Corrections and Limitations

What I had to fix or improve manually

Although AI generated useful starting points, the output still required manual checking. In several cases, I had to fix issues myself. These corrections were important because they show that I did not simply accept the AI output without understanding it. They also show how I detected subtle problems by testing the pages in the browser, comparing them with the rubric, and checking whether the generated solution really matched the assignment requirements.

Broken JavaScript after copy-paste

In one draft, JavaScript code was inserted incorrectly and part of the script appeared as plain text on the page instead of running. I detected the problem by opening the page in the browser, noticing that the interactive behaviour had stopped, and checking the script placement. I then restored the JavaScript file and confirmed that the script tag was loading the corrected file.

Missing features after code rewrites

During some rewrites, useful features such as learning resource links disappeared even though the page still looked visually complete. This was a subtle problem because the result looked polished at first glance. I detected it by comparing the updated page with the earlier working version and the rubric requirements, then restored the missing features manually.

Explanations that were too short

Some AI-generated explanations sounded smooth and correct, but they were too descriptive and did not go deep enough for an educational tutorial page. I detected this by reading the rubric carefully and noticing that the page needed to explain why best practices matter, not just define terms. I then expanded the explanations to make the tutorial more useful for learning.

Need for alignment with the brief

AI could generate impressive features, but I still had to check whether those features directly matched the assignment requirements. This was especially important for the quiz, reflection, and demonstration expectations.

Critical Evaluation

Reliability, risks, and ethical considerations

AI-generated code can look convincing even when it contains problems. Because of that, I could not treat the output as automatically trustworthy. I needed to test the code, compare it with the project brief, and confirm that it actually worked in the browser. This was especially important for dynamic JavaScript features such as loading JSON, handling events, and showing result feedback. One subtle example was when a generated version of the quiz validation blocked submission but did not clearly identify which questions were unanswered. I detected that gap by comparing the page behaviour against the rubric and then improved the validation so unanswered questions were visibly highlighted.

One reliability risk is that AI may produce code that is technically possible but not appropriate for the assignment. For example, it might add unnecessary complexity, remove an earlier feature during a rewrite, or generate a solution that does not follow the required workflow. Another risk is that AI can produce explanations that sound clear but are still too shallow for an assessment page that is supposed to teach users.

There are also ethical considerations. Using AI in coding can save time, but it can also create the false impression that the student understands more than they actually do if they submit code they cannot explain. To avoid that problem, I focused on asking for comments, reading the logic, testing the outputs, and making sure I could explain the code in a live demonstration. I also understand that AI use should be acknowledged honestly in the reflection and references.

Reliability Risk

AI output may look correct but still contain syntax errors, missing features, or poor alignment with the assignment brief.

Learning Risk

Relying on generated code without understanding it would make the live demonstration difficult and reduce real learning.

Ethical Risk

Submitting AI-generated code without acknowledgement or understanding would be academically irresponsible.

How I Responded

I reviewed outputs, tested pages in the browser, expanded weak explanations, restored missing features, and made sure I could explain the final code myself.

Learning Outcome

What I learned from using AI

The biggest lesson I learned is that AI is strongest when it is used as a guided assistant rather than as an automatic solution. It can speed up drafting, help generate ideas, and explain concepts in different ways, but it still needs human checking. In this project, the most useful workflow was: describe the goal clearly, examine the response, test it in the browser, then revise it until it matched both the brief and my own understanding.

I also learned that detailed prompts usually produce better results than vague ones. For example, asking for “more explanation”, “beginner-friendly comments”, “complete code”, or “interactive demonstrations” gave more useful output than simply asking for “a page”. This showed me that the quality of AI assistance depends strongly on the clarity of the instructions I provide.

Finally, I learned that using AI responsibly still requires effort. The tool can support productivity, but it does not remove the need for testing, debugging, editing, and critical reflection. In that sense, AI was most useful when it worked alongside my own judgement rather than instead of it.
